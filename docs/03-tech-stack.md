# Tech Stack

Concrete technology choices for the API client. The shell is **fixed as Wails** (Go backend + web frontend); this document records the module/library decisions inside that shell, why each was chosen, and the leading alternative that was rejected. This supersedes the earlier Tauri/Rust stack — the **frontend is unchanged** (Wails is webview-based, so the SolidJS app carries over untouched); everything backend moves from Rust crates to Go modules.

Verified current as of **July 2026**. Versions are pinned intentions, not lockfile truth — re-verify at `go.mod` bootstrap time.

Guiding constraints that drive nearly every choice below:

- **Lightweight + fast** — prefer small binaries and small JS bundles; avoid tens-of-MB blobs (V8, Monaco) unless there is no alternative. Wails rides the OS webview (WKWebView/WebView2/WebKitGTK) — no bundled Chromium.
- **Cross-platform-clean** — macOS first, but every choice must cross-compile to Windows/Linux without vendoring native binaries or fighting the toolchain. This favors **pure-Go** modules (no cgo) so `GOOS`/`GOARCH` cross-compilation just works — the exception is the WebKit binding Wails needs on macOS, which forces macOS runners for the darwin leg.
- **Git-friendly local storage** — plain-text, diffable files as the source of truth; the database is a disposable cache.
- **One headless execution engine** — the GUI, the MCP server, the CLI runner, and the k6 perf runner are all thin consumers of the same Go core package, so behavior is identical whether a human clicks, Claude Code calls over MCP, or CI runs it. Every outbound request passes through **one policy chokepoint at dispatch**.

---

## App shell decision — Wails v2 now, architect for v3

**Build on Wails v2 (stable); keep the Wails-specific binding layer thin so a later v2→v3 migration is a swap, not a rewrite.** Wails v3 is still **alpha** as of mid-2026 (`v3.0.0-alpha.10x` nightly pre-releases, no committed GA date) — its service/DI pattern, native multi-window, built-in updater, and new `wails3`/Taskfile build system are architecturally attractive but not worth eating alpha instability for a v1 ship. v2's API surface (single-window, context-threaded runtime calls) differs enough from v3 that the official migration guide exists precisely because it is not a drop-in. The core-engine bet (a UI-agnostic Go package consumed identically by GUI/MCP/CLI/k6-gen) is exactly what keeps the binding layer thin and the migration cheap regardless of where we start.

## Backend (Go) — core

| Layer | Chosen tech / module | Why | Alternative rejected |
|---|---|---|---|
| App shell | **Wails v2** (`github.com/wailsapp/wails/v2`) | Go backend + system-webview frontend → small binaries vs Electron; rides WKWebView/WebView2/WebKitGTK so no bundled Chromium. Go gives first-class `os/exec` (k6 sidecar), strong stdlib networking, and no separate Rust toolchain. Direct prior art: **Restmate**, a Wails-based REST client. | **Wails v3** (alpha — service pattern + multi-window + built-in updater are compelling but unstable; revisit at v3 beta); **Electron** (bundles Chromium, ~100MB+, higher RAM — contradicts "lightweight"); **Tauri** (prior stack — replaced by the Go direction). |
| Concurrency model | **Go goroutines + `context.Context`** | No async runtime to choose — Go's scheduler is built in. Every networking module below (`net/http`, `coder/websocket`, `grpc-go`) is context-native. One process owns all connections; the frontend never touches sockets. `context.Context` is the single cancellation idiom across all five protocols. | External actor/reactor frameworks (unnecessary — the stdlib scheduler and `context` cover it). |
| Backend→frontend streaming | **Wails `runtime.EventsEmit` as a wake-up signal, not a data pipe** — heavily batched/coalesced, with the frontend pulling full payloads via binding calls | Wails events have **no documented backpressure** and known drop/consistency issues under load (issues #2759, #2448). Treat events as a notification; cap emit rate (10–30Hz), coalesce server-side, and have the frontend pull authoritative history from a query binding. This satisfies "persist server-side as system-of-record; no lossy ring buffer." | Raw high-frequency `EventsEmit` per WS frame / gRPC message / SSE event / k6 point (no flow control → dropped data at high volume). |
| Large / binary payloads | **Local loopback HTTP handle** — write big bodies to a temp file or small embedded loopback server; pass only a URL/handle over the Wails bridge | Wails **auto-base64-encodes** binary crossing the Go↔JS boundary (~33% inflation + JSON serialization; open issue #3212, no zero-copy path). On Windows, WebView2's `postMessage` adds a large-payload latency cliff. Route big response bodies, gRPC blobs, file downloads, and k6 artifacts around IPC via `fetch()` on loopback. | Pushing large bodies through bindings/events (slow, memory-heavy, Windows-latency-exposed). |
| Session / cancellation | **`context.WithCancel` per request/stream**, held in a `Session{id, cancel, eventSink}` | One uniform cancel primitive across all five protocols. Every active request/stream carries a context derived from the same per-request cancellation scope (user Cancel button, collection-run abort, k6 stop). Honest per-protocol semantics: gRPC propagates `CANCELLED` to the server, not just a client-side give-up. | Ad-hoc per-protocol channel-drop / goroutine-kill (fragmented, no shared UI model, dishonest cancellation). |

## Backend (Go) — protocols

All five protocols share **one `crypto/tls` config path** (custom `RootCAs` + optional client identity via `Certificates`/`GetClientCertificate`) so TLS/mTLS/custom-CA is configured once per environment and honored everywhere. `crypto/tls` is stdlib — no OpenSSL linkage, fully cross-compilable; `GetClientCertificate` cleanly supports keychain-backed client certs. `InsecureSkipVerify` is a distinct, loudly-labeled per-environment toggle surfaced/logged at the same dispatch chokepoint as the policy gate.

| Layer | Chosen tech / module | Why | Alternative rejected |
|---|---|---|---|
| HTTP / HTTP/2 | **`net/http` (stdlib)** — own the `Transport`/`Client`, thin in-house wrapper | Both HTTP/1.1 and HTTP/2 built in (h2 auto-negotiated via ALPN). Zero dependency risk; full control over `Transport` for pooling/proxy/TLS and — critically for an *inspector* tool — wire-level detail (headers, timing, TLS handshake) via `net/http/httptrace`, which convenience wrappers hide. Per-request cancellation via `context.Context`. | **`resty` (resty.dev/v3)** (fluent convenience layer over `net/http`; skip it — a request/response inspector needs low-level introspection resty abstracts away; its native SSE client is noted but not enough to adopt it); **`isahc`/`ureq`-equivalents** (N/A — stdlib is the baseline). |
| HTTP/3 (QUIC) | **`github.com/quic-go/quic-go/http3`**, opt-in per-request | The only viable Go HTTP/3 library (used by Caddy/Cloudflare tooling); `http3.Transport` implements `http.RoundTripper`, so it plugs into the *same* `http.Client` — protocol selection is a config knob, not a separate code path. Ship as opt-in (default h2) because QUIC/UDP is often blocked on corporate networks. | Making h3 the default (network-blocked); hand-rolling QUIC (no reason — quic-go is the de-facto standard). |
| TLS backend | **`crypto/tls` + `crypto/x509` (stdlib)** | Portable and reproducible across macOS→Windows→Linux with no native linkage. mTLS via `tls.LoadX509KeyPair` / `GetClientCertificate` (dynamic, keychain-backed cert selection); custom CA via `x509.NewCertPool()` + `AppendCertsFromPEM` per environment; "disable SSL verification" via the explicit `InsecureSkipVerify` toggle. | **OpenSSL/native-tls** (per-OS linking pain, non-reproducible — Go's stdlib removes the need entirely). |
| WebSocket | **`github.com/coder/websocket`** (formerly `nhooyr.io/websocket`) | Actively maintained (Coder-backed since 2024); minimal API built natively around `context.Context` for read/write deadlines + cancellation — matches "honest per-protocol cancellation." Handles concurrent writes safely by design. Handshake is an HTTP upgrade → reuse headers/auth/subprotocols and the same TLS config for mTLS parity. | **`gorilla/websocket`** (**archived Dec 2022**, no security patches, predates `context` — do not adopt new); **`fastwebsockets`-equivalents** (lower-level/server-oriented; per-frame throughput isn't the bottleneck — the webview render is). |
| gRPC | **`google.golang.org/grpc`** + **`google.golang.org/protobuf/types/dynamicpb`** + **`github.com/jhump/protoreflect`** (reflection/dynamic invocation) + **`github.com/bufbuild/protocompile`** (runtime `.proto` compile) | Runs *fully dynamic* (no build-time `.proto`): discover schema via **server reflection** (`grpcreflect`, speaks v1 and v1alpha) or **imported `.proto`** compiled at runtime by pure-Go **`protocompile`** (successor to deprecated `protoparse`, no `protoc`/cgo); build/read messages by name with **`dynamicpb`** (JSON-in-editor → protobuf); invoke unary/streaming off a `MethodDescriptor` via **`grpcdynamic`**. **This exact stack is what k6 itself uses in production** — strong validation, and a genuine maturity edge over Rust's `protox`+`prost-reflect`. | **Codegen grpc-go stubs** (types generated at build time — useless for arbitrary runtime services); **vendored `protoc`** (ships a native binary per platform; `protocompile` removes the need). `connect-go` noted for the app's *own* HTTP surface, but target servers speak standard gRPC. |
| GraphQL | **`net/http` (reuse)** — thin client POSTing `{query, variables, operationName}`; **`github.com/hasura/go-graphql-client`** for subscriptions only | GraphQL is **not a transport** — a query is an HTTP POST, so reuse the HTTP client entirely. Introspection (`__schema { … }`) is just another POST — no schema library needed; parse the JSON result. Subscriptions run over WebSocket (graphql-transport-ws) via the hasura client, whose handshake/keepalive edge cases are worth not hand-rolling. | **`Khan/genqlient`** (compile-time codegen from a *known* schema + fixed query files — wrong shape for a tool where users paste arbitrary queries at runtime); `graphql-go/graphql` as an execution engine (we're a client, not a server). |
| Server-Sent Events | **Hand-rolled `text/event-stream` parser** on the `net/http` streaming body (`bufio.Scanner`) | ~100 lines to parse `event:`/`data:`/`id:`/`retry:`, and owning the loop lets us write straight into the persistence sink with natural backpressure — a slow consumer blocks the body read, which blocks TCP, exactly the backpressure-to-persistence the project requires ("no lossy ring buffer"). Reuses all TLS/proxy/auth config for free. | **`github.com/r3labs/sse/v2`** (fine, MPL-2.0, used by docker-compose — kept as the drop-in alt if we'd rather not own the loop); resty v3's `SSESource` (would pull in resty we otherwise skip); hand-parsing without persistence integration (misses the backpressure requirement). |
| Proxy | **`Transport.Proxy` (stdlib)** for HTTP/HTTPS; **`golang.org/x/net/proxy`** for SOCKS5 | `http.ProxyFromEnvironment` reads `HTTP_PROXY`/`HTTPS_PROXY`/`NO_PROXY` out of the box; per-environment proxy override is just a distinct `Transport`. `x/net` carries stdlib-equivalent governance for the SOCKS5 dialer. | Third-party proxy libraries (stdlib + `x/net` cover it). |

## Backend (Go) — scripting & execution engine

| Layer | Chosen tech / module | Why | Alternative rejected |
|---|---|---|---|
| JS scripting engine (pre/post-request, chaining, assertions) | **`github.com/grafana/sobek`** (goja fork), behind a `ScriptEngine` interface | **Pure Go, zero cgo** — cross-compiles to any `GOOS/GOARCH` (incl. universal macOS, Windows-from-macOS-CI) with no C toolchain, the decisive property here. Sobek is the Grafana/k6-maintained fork of `dop251/goja` (better-funded upstream than goja's single maintainer; the same lineage k6 uses since v0.52). Capability sandbox: a fresh `Runtime` has no `require`/`fs`/net by default — only what we bind. `Interrupt()` on a watchdog goroutine kills `while(true)`; `SetMaxCallStackSize` bounds recursion; `GOMEMLIMIT` is the process-level memory backstop. Fresh runtime per top-level script invocation for isolation. | **`dop251/goja`** (same API but single-maintainer bottleneck — use the Sobek fork); **`rogchap/v8go`** (cgo V8 — recreates the exact cross-compile/Windows-prebuilt trap the Go port exists to escape; no first-party Windows lib); **`robertkrimen/otto`** (ES5.1-only, effectively frozen). |
| `pm.*` / `bru.*` compatibility shim | **JS-source polyfills + narrow Go host bindings** loaded into each runtime at init | Eases migration from Postman/Bruno (a differentiator). `pm.environment/variables`, `pm.request/response`, Chai-style `pm.expect`, `CryptoJS`/`atob`/`btoa` are all pure-JS or trivial map bindings. **Deliberately excludes Node built-ins** (`fs`, `child_process`, `require`) — that's the vm2/NodeVM security-liability class Bruno had to walk back; goja/Sobek never had that surface, keep it that way. | Replicating Postman's Developer-Mode Node access (reintroduces the ambient-host attack surface for free). |
| Template / chaining resolver | In-house resolver on a **dependency DAG**, engine-backed; scripts are **node lifecycle hooks** on the same graph | `{{ response('LoginReq').body.token }}` refs create dependency edges; topological sort resolves order and auto-sends prerequisites. Scripts and native interpolation read/write **one shared variable store** so chaining behaves identically whether scripted or picked via UI. Pre-script + request + post-script **share one `context.Context` deadline** so a slow pre-script can't silently blow the node's timeout budget. | Standalone template lib divorced from scripting (forks resolution logic); separate script/native stores (UX inconsistency — the wrong kind of feature-count-over-coherence). |
| Execution core + **policy chokepoint** | **Headless `engine.Dispatch(ctx, spec)`** (no UI dependency), emits structured events | Single source of truth reused by GUI, MCP server, CLI runner, and k6-gen. **The policy gate lives inside `Dispatch`, not at the MCP boundary** — a human click, an MCP tool call, and a script's `pm.sendRequest` are indistinguishable to the policy layer, and none can bypass it (the script shim has no `net/http` bound to it, only `engine.Dispatch`). Emits `StepStarted`/`RequestSent`/`ResponseReceived`/`Captured`/`Assertion`/`Error` — GUI renders live, MCP serializes as tool results, perf runner aggregates. | Per-consumer execution paths (GUI ≠ MCP ≠ CLI → drift); gating at the MCP boundary only (script-/chain-initiated requests would bypass approval). |

## Backend (Go) — storage & secrets

| Layer | Chosen tech / module | Why | Alternative rejected |
|---|---|---|---|
| On-disk file format | **YAML**, one resource per file (`app.<type>.<slug>.yaml`) | Multiline **block scalars** keep request bodies / scripts / GraphQL queries readable in diffs; **comments** let users annotate requests. One-resource-per-file means two people editing two requests never touch the same file → git auto-merges. | **JSON** (single-line escaped `\n` bodies → unreadable diffs; no comments); **TOML** (array-of-tables collapses under nested heterogeneous auth/chain configs); monolithic `collection.json` (conflicts on every parallel edit). |
| YAML serializer | **`go.yaml.in/yaml/v3`** + a canonicalization layer | **Not `gopkg.in/yaml.v3`** — that path is archived; the spec-org-backed `go.yaml.in/yaml/v3` is the drop-in, API-identical successor (v4 exists but don't bind the on-disk schema to a library still taking breaking changes). `Marshal` already sorts map keys deterministically (diff-stable); a canonical writer adds explicit `orderKey` (reorder = 1-line diff, never array-position), force-quoted ambiguous scalars, LF + trailing newline. Model ordered data (headers, request lists) as explicit slices / `yaml.Node`, never `map` iteration order. Add a golden-file byte-identical CI test. | Archived `gopkg.in/yaml.v3` (unmaintained — reaching for it out of muscle memory is the trap); hand-rolled emitter (reinvents the encoder). |
| Local database | **`modernc.org/sqlite`** (pure-Go) + WAL + FTS5 (`-tags sqlite_fts5`) | DB is a **disposable cache**: history, response blobs, k6 run results, FTS5 search index, cookie jars, UI state — byproducts of *running*, rebuildable from files. **Pure Go = no cgo**, so cross-compile + notarize stays clean (the single cleanest "risk removed" of the Rust→Go move); the ~1.5–2× throughput gap vs the C engine is irrelevant at local-cache scale. FTS5 is **not default-on** — pass `-tags sqlite_fts5`. | **`mattn/go-sqlite3`** (cgo → per-target C cross-compiler, Xcode/`dsymutil` pain, static-link flags — exactly the toolchain fragility pure-Go avoids); **`sqlx`/async ORMs** (overkill for a local-only file DB). |
| OS keychain | **`github.com/zalando/go-keyring`** — stores only a per-install master key / secret refs | macOS Keychain / Windows Credential Manager / Linux Secret Service (libsecret) via one small API (`Set`/`Get`/`Delete`). **Version-pin note:** verify the macOS backend calls Keychain Services directly rather than shelling out to `/usr/bin/security` (issue #110 — the shell-out is a documented hardening concern); prefer the direct-API path at the pinned version. | Node `keytar` bridges (adds a JS dependency); Linux `keyutils` as default (session-scoped, doesn't survive reboot — fall back to an argon2 passphrase-derived key on headless/CI Linux with no Secret Service). |
| Secret encryption | **XChaCha20-Poly1305** (`golang.org/x/crypto/chacha20poly1305`) on values; key lives in the OS keychain | Two-tier: the keychain holds one small master/vault key; that key `Seal`/`Open`s any secret material that must be cached outside the keychain (disposable SQLite rows, encrypted exports/backups). **XChaCha20's 192-bit nonce** makes random per-encryption nonces safe (no nonce-reuse footgun). Official `x/crypto` — no third-party crypto risk. Ciphertext is safe to commit → encrypted-but-shareable environments. | **AES-256-GCM** (valid but 96-bit nonce needs careful management); plaintext-on-disk (leaks via git/cloud sync); persisting the key alongside ciphertext (defeats the point). |
| Git integration | **`github.com/go-git/go-git/v5`** (pure Go), fall back to shelling to `git` for exotic remotes | In-app git UI (status/diff/commit/pull/push) over the plain-YAML workspace — **no dependency on the user having `git` installed** (they may treat this like Postman). Pure Go = trivially cross-compiled, matches the no-cgo story. Collection repos are small (tens–low-thousands of files), so go-git's known large-repo memory/perf ceiling won't bite. For the rare push-credential / exotic-remote / proxy case, `exec.LookPath("git")` and delegate — same arm's-length pattern as k6. | **v6** (still alpha — revisit for its wildmatch/status improvements once stable); **always shelling to system `git`** (dependency on a `git` binary + PATH); proprietary cloud sync (deferred; layer on the same files later, opt-in). |
| Order keys | **Vendored fractional-index** (port of `rocicorp/fractional-indexing`; `rocicorp/fracdex` upstream if freshness verifies) | Reordering one request touches exactly **one** YAML file (merge-safe, git-diffable) vs an integer `position` that renumbers/diffs every sibling. Algorithm is ~150–300 LOC and stable — vendoring removes a supply-chain unknown for a component this load-bearing to the merge story. Store as an explicit `orderKey` string field per resource. | Integer position columns (renumber-on-reorder → huge diffs, merge conflicts); depending on `fracdex` without checking its maintenance freshness. |

## Backend (Go) — performance/load testing (k6)

| Layer | Chosen tech / approach | Why | Alternative rejected |
|---|---|---|---|
| Load generator | **k6** (Grafana), bundled as an **arm's-length CLI sidecar**, invoked via `os/exec` — **never** imported in-process | Single static Go binary → ideal sidecar. Core engine owns k6-script *generation* (JSON-encoded literals, not string concat, to avoid injection) so the generated test reuses the same request→HTTP mapping as normal dispatch. Spawn with `exec.CommandContext`; stream **live NDJSON** (`--out json=-`) parsed with `bufio.Scanner` (raise the buffer past 64KB) and **aggregated server-side into 250ms–1s buckets** before emitting Wails events (k6 point volume scales VUs×iteration-rate — the most likely place to blow a naive channel); persist raw NDJSON + buckets to the run's store ("no lossy ring buffer"). End-of-test `handleSummary` + `--summary-export` give the canonical report; **exit code 99 = thresholds failed** → deterministic `{passed, failedThresholds}`. SIGINT = graceful (still runs `handleSummary`). | **Importing `go.k6.io/k6/...` in-process / `xk6` custom build** (produces an **AGPL derivative work covering the whole binary** — see below; the sharpest trap on Go, since `go get` just works); computing p95 from the live stream in JS (let k6 compute percentiles in `handleSummary`). |

### ⚠️ k6 AGPLv3 licensing consideration — **sharper on Go than it was on Rust**

k6 is licensed **AGPL-3.0** (Grafana relicensed from Apache-2.0 in 2021; still AGPL through k6 2.0, May 2026). On the old Tauri stack, a Rust host + Go-only k6 created a language barrier that made accidental in-process embedding awkward. **On Wails, both the app and k6 are Go — `import "go.k6.io/k6/js/modules"` compiles cleanly, autocompletes, and looks like an ordinary dependency add.** That ease-of-reach is the trap, and it is new and sharper here.

1. **Keep the boundary a CLI child process — always.** Invoking stock k6 as a separate executable is *mere aggregation* / arm's-length and does **not** force the app's own source under AGPL. Statically linking/importing k6's Go packages produces **one combined executable** that AGPL §5/§13 treat as a single work — copyleft extends to the *entire* app (engine, MCP server, GUI glue — everything linked in). **Do not** `import go.k6.io/k6/*` anywhere in the module graph, `go:embed` k6's source, `xk6`-build, or vendor k6 "just for one helper."
2. **Do not follow Wails's own idiomatic sidecar advice for k6.** Wails has no `externalBin` equivalent; the maintainer's recommended pattern is `go:embed` the binary bytes into the single Go executable — which is precisely the "combined/distributed as one work" posture to avoid. Instead, stage a **separate, unmodified k6 binary** into the `.app` bundle's `Contents/Resources/` (or `Contents/MacOS/`) via a build-pipeline step (Taskfile/Makefile after `wails build`), a file the user could delete/replace independently.
3. **Bundle stock, version-pinned k6, and ship Corresponding Source — not just a pointer.** Distributing the k6 binary triggers AGPL §6/§13 source obligations *regardless of network use*. Build k6 from the pinned tag in CI, publish the resulting source tarball next to each app release, and record version + build flags + source location in the release manifest (must survive deprecating old releases). Attribution screen: AGPLv3 text + copyright + durable source pointer.
4. **AGPL §13 (network use) is a red herring for a desktop app, not the exemption people think.** §13 is an *additional* SaaS trigger on top of ordinary GPL copyleft — it does **not** mean local desktop use is exempt. Shipping a compiled binary that statically links AGPL code to end users **is** distribution under §5 and triggers copyleft on its own. "It's just a desktop app" does not get you out.
5. **MCP interaction is fine** — the app triggering a local k6 child process is the same aggregation boundary.

This is not legal advice — the owner should confirm with counsel (even k6's own maintainers call the process-boundary question legally unsettled); shipping Corresponding Source is the belt-and-suspenders call regardless.

## Frontend (web)

The frontend is **unchanged from the prior stack** — Wails is webview-based, so the SolidJS app carries over untouched. It is a consumer of backend bindings + event streams; it holds **no protocol logic**. Every choice is weighted toward small JS bundle and fast render (the "lightweight + fast GUI" thesis).

| Layer | Chosen tech / library | Why | Alternative rejected |
|---|---|---|---|
| UI framework | **SolidJS + TypeScript** (Vite) | Fine-grained reactivity with **no virtual DOM** → updates only the exact DOM node that changed, no reconciliation. Decisive for a streaming-heavy tool (live WS/gRPC frames, k6 metric buckets, live byte counters): React would re-render/reconcile subtrees per batch where Solid mutates one text node. Smallest runtime → directly serves the "lightweight + fast GUI" thesis. CodeMirror 6 / uPlot / TanStack all work via Solid adapters. | **React** (largest talent pool, but VDOM reconciliation is exactly the wrong cost profile for high-frequency streaming — mitigable with virtualization but not free; fallback if hiring/ecosystem ever outweighs render cost). **Svelte** (also compiled/fast, but Solid's signal model maps more cleanly onto "subscribe a DOM node to a stream"). **Angular** (too heavy). |
| Code / body editor | **CodeMirror 6** | Tree-shakeable, ~50–150KB for a working editor vs Monaco's 2–5MB gzipped. Modular language packages (JSON, GraphQL, JS for scripts). Replit/Sourcegraph migrated *off* Monaco to CodeMirror for exactly this. Directly serves "lightweight". | **Monaco** (VS-Code-grade IntelliSense but 2–5MB bundle — 40% of a page's JS; overkill for request bodies/scripts and fights the bundle-size goal). |
| Live/perf charts | **uPlot** | ~50KB, Canvas 2D, purpose-built for **streaming time series** — 60fps live updates at ~10% CPU / 12MB RAM where Chart.js/ECharts use 40–70% CPU and 77–85MB. Exactly the k6 live-metrics workload (throughput / p95 / error-rate / VUs). | **Chart.js / ECharts / Recharts** (far heavier CPU+RAM at streaming rates; Recharts' SVG DOM chokes on high-frequency updates). WebGL libs (overkill — no waveform-scale data here). |
| State management | **Solid stores + signals** (+ **TanStack Query** Solid adapter for async/command state) | Solid's built-in signals/stores are the idiomatic, zero-extra-dependency store for fine-grained reactivity — UI state (open tabs, panes, selection, editor buffers). Live event streams from Wails events push straight into signals so only bound DOM nodes update; authoritative payloads are pulled via bindings. TanStack Query (Solid adapter) manages backend binding results — caching + invalidation, keeping files/DB in Go as the single source of truth. | **Zustand/Redux/MobX** (React-oriented external stores — unnecessary once Solid's signals cover fine-grained state natively); **Context-only** (coarse-grained → re-render storms at this app's update frequency). |
| Styling | **Tailwind CSS** (+ CSS variables for theming) | Utility-first → fast iteration, no runtime CSS-in-JS cost, tiny production CSS after purge. CSS variables drive the theme system (Dark/Light + community themes). | **CSS-in-JS** (styled-components/Emotion — runtime cost, larger bundle, contradicts "fast"); heavy component kits like MUI (bundle bloat + opinionated look). |

## Build, packaging, signing, auto-update

| Concern | Chosen tech / approach | Why | Alternative rejected |
|---|---|---|---|
| Bundler | **Wails CLI (`wails build`)** + a hand-rolled sidecar-staging step | Wails v2 produces `.app`/`.dmg` (macOS), NSIS/MSI (Windows), and Linux packages from one config. **The macOS leg needs real macOS runners** (WebKit binding uses cgo — you can't cross-compile *to* darwin the pure-Go way). Universal (arm64+x86_64) via lipo, wired explicitly into CI — the k6 sidecar needs the same universal/per-arch treatment or Rosetta silently eats the mismatch on Intel. | Hand-rolled packaging per OS (reinvents most of what Wails does). |
| Sidecar staging | **Build-pipeline copy** (Taskfile/Makefile) of the pinned k6 binary into `Contents/Resources/` after `wails build` | Wails has **no `externalBin` feature**, and its recommended `go:embed` pattern is the AGPL trap (see k6 §). Hand-roll a per-arch copy step so k6 stays a separate on-disk file the user can replace — a build-tooling gap vs Tauri, not a legal blocker. | `go:embed`-ing k6 into the single Go binary (creates the combined-work AGPL problem); committing k6 binaries to the repo (bloats git; no clean version pin). |
| macOS code signing + notarization | **`codesign` + `notarytool` + `stapler`** (owned shell/Taskfile step) + Apple Developer ID; **sign the k6 sidecar + bridge individually, bottom-up** | `wails build` produces the bundle but does **not** sign/notarize — that's a wrapper step you own. **Every nested Mach-O must be signed individually** — don't rely on `--deep`. Hardened runtime (`--options runtime`) is mandatory; entitlements are **not inherited** by the spawned k6 process, so k6 needs its own — likely `com.apple.security.cs.allow-jit` (and possibly `allow-unsigned-executable-memory`) for its sobek JS VM; validate empirically against the shipped k6 binary. `get-task-allow` must be absent in release. `spctl -a -vvv --type execute` on the final bundle as a CI pre-release check. | Shipping unsigned/ad-hoc-signed (Gatekeeper blocks); relying on `--deep`; signing only the main binary and forgetting the sidecar (passes submission, fails Gatekeeper at first launch on the user's machine). |
| Windows / Linux signing | Authenticode (Windows) / standard repo signing (Linux) — deferred | k6.exe needs the same Authenticode signing to avoid SmartScreen/AV flags. Linux has no central signing gate; AppImage bundles k6 in one file, deb/rpm/Flatpak update via package managers. Wails's Windows installer tooling is more mature than its macOS story. | — |
| Auto-update | **`github.com/minio/selfupdate`** as the mechanics layer + a self-hosted static JSON manifest (GitHub Releases / S3), swapping a **fully codesigned + notarized new `.app`** (not binary-patching in place) | **Weaker story than Tauri** — this is a genuine regression to flag and budget for. Wails v2 has **no built-in updater**; Wails v3's is alpha and tied to an alpha framework. `minio/selfupdate` (Apache-2.0, MinIO-proven) gives license-clean signature-verified self-replace decoupled from Wails's churn. **Critical macOS caveat:** bsdiff-patching a signed Mach-O invalidates its signature → Gatekeeper blocks it, so **swap a fully re-signed bundle** rather than patch in place (costs bandwidth, keeps Gatekeeper valid). | **Wails v3 built-in updater** (alpha-on-alpha; re-validate every upgrade + audit its signature path before trusting it — reasonable to prototype, risky as sole production mechanism); **Sparkle** (most mature/trusted on macOS but ObjC/Swift → a cgo bridge + a *second* updater codepath since it's macOS-only); **no auto-update** (poor UX for a shipped desktop app); **custom updater** (the one place "custom" is genuinely dangerous — a weak signature check is an RCE vector). |
| Sidecar acquisition | **`download-k6.sh`** build step: fetch pinned k6 releases per target triple, verify checksums, stage into the bundle | Keeps AGPL "corresponding source" trivial (pinned upstream tag, unmodified) and each build deterministic. | Committing k6 binaries to the repo (bloats git; no clean version pin); building a modified k6 (risks an AGPL derivative). |

## Testing

| Layer | Chosen tech / approach | Why | Alternative rejected |
|---|---|---|---|
| Go unit / integration | **built-in `go test`** + **`github.com/stretchr/testify`** (assert/require/mock); **`net/http/httptest`** for HTTP fixtures | Standard, minimal tooling. `httptest.Server` stands up a local mock to exercise the `net/http`/execution-engine paths deterministically. | Go golden / property | **golden-file snapshot** (via `go test -update` convention, e.g. `goldie`) for the YAML canonical writer + k6 script codegen; **`testing/quick`** or a fuzz target for the template/DAG resolver | The canonical serializer and script codegen are exactly what golden tests protect (catch diff-churn regressions — pairs with the byte-identical CI check on YAML output). | Concurrency safety | **`go test -race`** in CI on the streaming/event paths | Wails's own events system has a history of data races under concurrent emit/register (issue #2448); the goja/Sobek `Interrupt` path has historical race/deadlock reports (#16, #97). Run the race detector on the backpressure/streaming and script-interrupt code — stress N concurrent runtimes running `while(true)` with a watchdog `Interrupt` and assert all return within the deadline. | Trusting the docs / no race detector (the exact bug class these libraries have shipped historically). |
| Frontend unit / component | **Vitest** + **Solid Testing Library** | Vite-native (shares the app's transform pipeline), fast; Solid Testing Library tests components against user-facing behavior (same `@testing-library` API family, Solid renderer). | Jest (slower, separate transform config from Vite); React Testing Library (wrong renderer for a Solid app). |
| End-to-end (drives the real Wails app) | **WebDriver against the running Wails webview** (WebdriverIO), acknowledged as **less paved than Tauri's `tauri-driver`** | Wails has **no first-party e2e harness** equivalent to `tauri-driver` — another gap to flag. Drive the actual WKWebView/WebView2 + Go backend together (real bindings, real sidecars) via WebDriver where the platform webview supports it; on macOS this is the weakest leg, so lean harder on the self-dogfooding path below. | Playwright/Cypress alone (can't drive the native Wails shell — only a browser); manual QA only (doesn't scale). |
| Self-dogfooding | Use the app's **own CLI/MCP runner** for API contract tests in CI | The headless engine + assertion engine is itself a CI test runner (non-zero exit on failure, JUnit/JSON reporters) — the app tests itself and demonstrates the feature. Partly compensates for the thinner Wails e2e story. | External Newman/Bruno CLI in our own CI (we're building the better one). |

## Summary — the load-bearing decisions

1. **Wails v2 (stable), not v3 (alpha)** — keep the Wails binding layer thin so a v2→v3 migration is a swap; the UI-agnostic Go core engine is what makes that cheap.
2. **`crypto/tls` (stdlib) as the single TLS path** feeds all five protocols; one cert config per environment, `GetClientCertificate` for keychain-backed mTLS, `InsecureSkipVerify` a loud opt-in.
3. **One `context.Context` cancellation idiom** unifies HTTP-streaming / WS / gRPC-bidi / SSE / k6 metrics under one Cancel button — with *honest* semantics (gRPC propagates `CANCELLED` server-side).
4. **gRPC is fully dynamic** — grpc-go + `dynamicpb` + `protoreflect` + `protocompile` (pure-Go, no vendored `protoc`); this is the exact stack k6 uses in production and a real maturity edge over Rust's protox.
5. **GraphQL is not a transport** — `net/http` + a WS subscription client; no codegen client.
6. **Sobek (goja fork), not v8go** — pure Go, zero cgo, sandboxable (`Interrupt` timeout + `SetMaxCallStackSize` + `GOMEMLIMIT`), cheap fresh-runtime-per-invocation; avoids v8go's cgo/Windows-prebuilt trap.
7. **`go.yaml.in/yaml/v3` + one-file-per-resource + `modernc.org/sqlite` (pure-Go) cache** — files are truth and diff cleanly; the DB is disposable and rebuildable, with no cgo anywhere in the storage layer.
8. **Secrets in the OS keychain (`zalando/go-keyring`) + XChaCha20-Poly1305** — the master key never leaves the keychain; only encrypted-but-shareable ciphertext touches git.
9. **Official `modelcontextprotocol/go-sdk`, embedded Streamable-HTTP server** on the running app (loopback + bearer token), so Claude Code drives live UI state; **the policy gate lives in `engine.Dispatch`**, identical for clicks, MCP calls, and script-initiated requests — nothing bypasses approval.
10. **k6 as a stock, pinned, CLI-invoked AGPL sidecar** — the arm's-length boundary keeps the app's source out of AGPL; the trap is *sharper on Go* (both are Go, `go get` just works, and Wails's idiomatic `go:embed` is the wrong answer), so stage a separate binary and ship Corresponding Source.
11. **Weaker Wails auto-update + e2e story** — `minio/selfupdate` with full-bundle re-sign/notarize (not in-place patching) for updates; lean on self-dogfooding CI to offset the missing `tauri-driver` equivalent. Both are budgeted regressions vs the Tauri baseline.
12. **Frontend unchanged — SolidJS + CodeMirror 6 + uPlot + Tailwind** — no-VDOM fine-grained reactivity plus every frontend pick optimizes bundle size and streaming render for the "lightweight + fast GUI" thesis; Wails being webview-based means the port cost here is zero.
